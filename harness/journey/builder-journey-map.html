NodeKit · Builder Journey J0 · Re-anchor Map

From an idea to a live, improving product — one case, five handoffs.

NodeKit has powerful engines: graph-hop, Atlas, create/adopt, the frontend tournament, NodeSlide/NodeVideo, the EASE gate, the evolution ledger, the harness gym. What it lacks is the vehicle — a single case that threads one caseId through every stage, each with an explicit handoff artifact and a receipt proving the handoff was legitimate.

Thesis Gates are stage-exit conditions, not the product. The engines already gate their own exits; the journey is the connective layer that carries the builder between them — fail-closed at every seam.
01

DECIDE

Select the user, the uncertainty, the read-only wedge.

Handoff artifact OpportunityContract
Done in repo
  • Research engines: graph research / query / gaps
  • Atlas retrieval: atlas search / recipe / preview
  • Domain-blank factory: create refuses presets
The gap
  • Nothing freezes the product boundary before scaffold
  • No record of what was not chosen or what the app may not do
  • The missing primitive this J0 builds.
02

BUILD

One golden journey against one frozen fixture.

Handoff artifact BuildEvidencePack
Done in repo
  • create / adopt scaffolding
  • Frontend tournament: frontend plan / benchmark / repair
  • Builder-gym: frozen fixtures + verdicts + locks
The gap
  • No single artifact binding {fixture + golden run + receipt}
  • Gym emits verdicts, but isn't wrapped as a stage handoff
03

EXPLAIN

Concise deck + short video + demo script from real evidence.

Handoff artifact StoryPack
Done in repo
  • Present skill + change-story contract
  • NodeSlide / NodeVideo transports
  • presentation-pipeline deck & media proof
The gap
  • No StoryPack tying deck+video+script to the BUILD evidence
  • Pipeline gates a campaign, not a per-case stage
04

LAUNCH

Publish the exact tested revision + one target submission.

Handoff artifact LaunchManifest
Done in repo
  • Submission gate + prepare / evaluate scripts
  • npm packaging + install-proof
  • Exact-revision binding: commit + source hash
The gap
  • No manifest binding tested revision → one submission
  • No deployment receipt as the stage exit
05

LEARN

Real use → one friction point → one response proposal.

Handoff artifact ObservationPack
Done in repo
  • Evolution ledger: record / verify / materiality
  • Harness gym: baseline / propose / promote
  • Friction telemetry captured at scaffold time
The gap
  • No pack shaping use into {friction, response}
  • Ledger records changes, not a Learn → Decide loop
DECIDE→ BUILD→ EXPLAIN→ LAUNCH→ LEARN ↶ back to DECIDE or BUILD A stage is complete only when its refs are non-empty AND a receipt binds it — advancing without the handoff artifact throws or returns blocked.
Worked example

Verified Weekly Salon Brief

the salon slice, as a concrete OpportunityContract instantiating DECIDE

The Decide→Build seam fails when a strong technical agent makes product decisions while coding. The OpportunityContract is the approved boundary it builds against instead of improvising — recording the user, the frozen wedge, and the authority limits so scope cannot be re-decided at the keyboard.

UserOwner of a one-location salon.
ProblemCan't tell during the week whether higher sales are producing higher profit.
WedgeA read-only weekly profit health check from uploaded files.
Primary jobUnderstand estimated operating profit and resolve the few items blocking verification.
Inputsbank_csv · square_export · payroll_summary · receipt_images
Primary artifactA source-linked Verified Weekly Salon Brief.
Authority limits — thin scope: one salon one week CSV / file uploads no live banking no accounting write no tax advice
Canonical model: nodekit.builder-case/v1 Missing primitive: nodekit.opportunity-contract/v1 Built on Caseflow — case / run / stage / artifact / receipt
