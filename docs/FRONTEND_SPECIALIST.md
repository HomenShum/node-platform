# NodeKit Frontend Specialist

NodeKit routes frontend implementation as a bounded specialist role. The specialist receives a protected product-design contract; it does not choose the primary user, primary job, canonical workflow, data authority, permission boundaries, completion criteria, or final verdict.

## Required pipeline

```text
Product design contract
-> evidence-ranked route
-> three materially different rendered directions
-> browser, accessibility, overflow, and visual checks
-> independent blind comparison
-> bounded repair
-> fresh-user canary
-> NodeProof and human approval
```

The route starts unprofiled. A model may become preferred only when its requested and resolved identities are supported by scoped Model Intelligence evidence. Model revision and tool-surface changes invalidate the preference.

## Commands

```bash
nodekit frontend init
nodekit frontend plan --contract harness/frontend/product-packets/product.yaml
nodekit frontend directions --plan harness/frontend/receipts/<plan>.json
nodekit frontend benchmark --manifest harness/frontend/benchmark.json
nodekit frontend repair --benchmark harness/frontend/benchmark.json
nodekit frontend canary --receipt harness/frontend/canary.json
```

The three required archetypes are collaborative workspace, artifact studio, and a domain-native interpretation. Each must render desktop first arrival, active workflow, proposal review, mobile artifact, mobile agent, and mobile review states.

No tournament, repair, or canary grants deployment authority. NodeProof and an authorized human retain that decision.
